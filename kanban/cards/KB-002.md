# KB-002 — Ansible Playbook: Configuração do OS e K3s

## Status
- **Coluna atual:** Backlog
- **Prioridade:** Alta
- **Criado em:** 2026-04-20

## Descrição

Criar o playbook Ansible responsável por preparar o sistema operacional e instalar o K3s. Este é o coração do `make install` — deve ser 100% idempotente e reproduzível.

Organizado em roles separadas para manter responsabilidades claras e facilitar manutenção futura.

## Critérios de Aceite

- [ ] Playbook idempotente (rodar 2x não causa erros nem mudanças desnecessárias)
- [ ] Role `common` executada com sucesso: OS configurado, swap desabilitado, timezone correto
- [ ] Role `k3s` executada com sucesso: K3s instalado, serviço ativo, Traefik e ServiceLB nativos desabilitados
- [ ] Role `sealed-secrets` executada: chave privada restaurada (se existir) ou exportada (se novo cluster)
- [ ] Kubeconfig copiado para `~/.kube/config` com permissões corretas para o usuário `elis`
- [ ] Testado em máquina limpa (Ubuntu 24.04)
- [ ] Testado idempotência: segunda execução sem erros nem mudanças desnecessárias

## Estrutura no Repositório

```
ansible/
├── playbooks/
│   └── site.yml                  # Playbook principal — chama as roles em ordem
├── roles/
│   ├── common/                   # Configurações base do OS
│   ├── k3s/                      # Instalação e configuração do K3s
│   └── sealed-secrets/           # Backup/restore da chave privada
└── inventory/
    ├── hosts.yml                 # Define o host alvo (cusin / 192.168.31.55)
    └── group_vars/
        └── all.yml               # Variáveis globais configuráveis
```

## Detalhamento das Roles

### Role: `common`
Configurações base do OS necessárias para o Kubernetes funcionar corretamente:

| Configuração | Valor | Motivo |
|---|---|---|
| `vm.swappiness` | `0` | K8s não funciona bem com swap |
| `fs.inotify.max_user_instances` | `512` | Evita erro com muitos pods/watchers |
| `fs.inotify.max_user_watches` | `524288` | Idem |
| Swap | Desabilitado permanentemente | Requisito do K3s |
| Timezone | `America/Sao_Paulo` | Logs com horário correto |
| Pacotes base | `curl`, `git`, `make` | Necessários para o bootstrap |

### Role: `k3s`
Instalação e configuração do K3s:

| Parâmetro | Valor |
|---|---|
| Versão | `v1.32.3+k3s1` (fixada em `all.yml`, atualizável) |
| Modo | Single-node (server sem agent separado) |
| Traefik nativo | **Desabilitado** (`--disable=traefik`) |
| ServiceLB nativo | **Desabilitado** (`--disable=servicelb`) |
| Container runtime | containerd (padrão do K3s) |
| Kubeconfig | Copiado para `~/.kube/config` do usuário `elis` |

### Role: `sealed-secrets`
Gerência da chave privada do Sealed Secrets para reproduzibilidade:

```
Se ~/.homelab/sealed-secrets-key.yaml existir:
    → Restaura a chave no cluster (rebuild de hardware)
Se não existir:
    → Aguarda o Sealed Secrets ser instalado pelo ArgoCD
    → Exporta a chave gerada para ~/.homelab/sealed-secrets-key.yaml
```

## Variáveis Configuráveis (`all.yml`)

```yaml
k3s_version: "v1.32.3+k3s1"
k3s_user: "elis"
timezone: "America/Sao_Paulo"
sealed_secrets_key_path: "~/.homelab/sealed-secrets-key.yaml"
```

## Notas Técnicas

- O Ansible roda localmente na própria máquina (`connection: local`) — não há máquina de controle separada
- A chave privada do Sealed Secrets **nunca entra no repositório Git**
- O kubeconfig gerado pelo K3s fica em `/etc/rancher/k3s/k3s.yaml` (root) — o Ansible copia e ajusta permissões para o usuário
