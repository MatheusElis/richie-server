# KB-026 — Integrar apps existentes com Authentik (Forward Auth)

## Status
- **Coluna atual:** Done
- **Prioridade:** Média
- **Criado em:** 2026-04-22
- **Concluído em:** 2026-04-23

## Descrição

Adicionar o middleware ForwardAuth do Authentik nos ingresses das aplicações já em produção: Glance e pgAdmin. Elimina acesso não autenticado e centraliza o controle de acesso via Authentik.

## Detalhes Técnicos

### Apps afetadas

| App | Namespace | Ingress | Método |
|---|---|---|---|
| Glance | glance | `home.bisnaguete.xyz` | Forward Auth puro (sem auth própria) |
| pgAdmin | pgadmin | `pgadmin.bisnaguete.xyz` | Forward Auth + Proxy auth (`X-Authentik-Email`) |

### Configuração no Authentik (via Blueprints)

Abordagem **domain-level** — um único Provider Proxy cobre todas as apps em `*.bisnaguete.xyz`.

Configuração declarativa via **Blueprints** (ConfigMap montado no worker):
- `apps/authentik/blueprints-configmap.yaml` — contém dois blueprints:
  - `forward-auth-domain.yaml`: ProxyProvider (forward_domain mode) + Application + Outpost binding ao embedded outpost
  - `argocd-oidc.yaml`: OAuth2Provider + Application para ArgoCD OIDC

Para proteger uma nova app, basta adicionar a annotation no ingress — sem criar nada no Authentik.

### Middleware Traefik

Arquivo: `apps/authentik/traefik-middleware.yaml`
- Tipo: `traefik.io/v1alpha1` Middleware ForwardAuth
- Endereço: `http://authentik-server.authentik.svc/outpost.goauthentik.io/auth/traefik` (porta 80, **não** 9000)
- Headers propagados: `X-authentik-username`, `X-authentik-email`, `Remote-User`, etc.

> **Atenção:** O Service `authentik-server` expõe porta 80 → targetPort 9000. O middleware deve usar porta 80 (padrão).

### Alterações realizadas

**Glance (`apps/glance/ingress.yaml`):**
- Adicionada annotation: `traefik.ingress.kubernetes.io/router.middlewares: authentik-authentik@kubernetescrd`

**pgAdmin (`apps/pgadmin/ingress.yaml`):**
- Adicionada mesma annotation de Forward Auth

**pgAdmin (`apps/pgadmin/deployment.yaml`):**
- `PGADMIN_CONFIG_AUTHENTICATION_SOURCES`: `['webserver']`
- `PGADMIN_CONFIG_WEBSERVER_REMOTE_USER`: `'X-Authentik-Email'` — header com email do usuário Authentik
- `PGADMIN_CONFIG_WEBSERVER_AUTO_CREATE_USER`: `True` — cria usuário automaticamente
- `PGADMIN_CONFIG_ENHANCED_COOKIE_PROTECTION`: `False` — necessário em Kubernetes (IPs dinâmicos)

> **Importante:** Variáveis de configuração do pgAdmin Docker devem usar prefixo `PGADMIN_CONFIG_`, e valores string precisam de aspas Python internas (`"'valor'"`).

**pgAdmin (`apps/pgadmin/servers-configmap.yaml`):**
- Adicionado `"Shared": true` no server PostgreSQL — torna visível para todos os usuários (inclusive os criados via webserver auth)

**Middleware (`apps/authentik/traefik-middleware.yaml`):**
- Corrigida porta de `:9000` para porta padrão (80) — compatível com o Service Kubernetes

### Lições aprendidas

1. **Service vs Pod port:** O middleware deve usar a porta do Service (80), não a porta do container (9000)
2. **pgAdmin Docker envvars:** Usar `PGADMIN_CONFIG_*` para sobrescrever `config.py`. Variáveis sem prefixo são ignoradas
3. **pgAdmin string values:** Precisam de aspas Python: `"'REMOTE_USER'"` (YAML externo + Python interno)
4. **Header email vs username:** `Remote-User` contém username (`akadmin`), `X-Authentik-Email` contém email. pgAdmin precisa de email
5. **Shared servers:** Usar `"Shared": true` no `servers.json` para que servers sejam visíveis por todos os usuários
6. **Outpost binding:** O Blueprint precisa vincular explicitamente o provider ao embedded outpost — sem isso o outpost não serve o proxy
