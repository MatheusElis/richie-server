## Descrição

Instalar e configurar o Sealed Secrets para permitir que secrets do Kubernetes sejam armazenados com segurança no repositório Git. A estratégia adotada usa um par de chaves **gerado manualmente uma única vez** — nunca pelo cluster — eliminando dependência circular no bootstrap e garantindo reproduzibilidade total.

## Critérios de Aceite

- [ ] Par de chaves gerado manualmente e armazenado em `~/.homelab/sealed-secrets-key.yaml`
- [ ] Ansible restaura a chave privada no cluster antes do ArgoCD ser instalado

- [ ] `kubeseal` CLI instalado automaticamente pelo `make deps`
- [ ] Todos os SealedSecrets do repositório encriptados com a chave gerada

## Estratégia de Chaves

A chave é **gerada manualmente uma vez** pelo usuário, nunca pelo cluster. Isso elimina a dependência circular do primeiro setup e facilita que outros usuários adotem o setup.

### Primeiro setup (geração das chaves)
```
make gen-keys
    └── Gera par de chaves RSA
        └── Chave privada → ~/.homelab/sealed-secrets-key.yaml  (NUNCA no Git)
        └── Chave pública → ~/.homelab/sealed-secrets-cert.pem  (pode ser compartilhada)
            └── Usar a chave pública para encriptar todos os SealedSecrets do repositório
```

### make install (qualquer execução)
```
Ansible detecta ~/.homelab/sealed-secrets-key.yaml
    └── Importa a chave privada no cluster (antes do ArgoCD)
        └── ArgoCD bootstrap
            └── ArgoCD instala Sealed Secrets controller (wave 0)
                └── Controller usa a chave já importada
                    └── ArgoCD aplica SealedSecrets (waves 1+) → decriptados ✅
```

### Rotação de chaves (chave perdida ou novo usuário)
```
make rotate-keys
    └── Gera novo par de chaves
        └── Re-encripta todos os SealedSecrets do repositório com a nova chave pública
            └── Commit automático com os novos SealedSecrets
                └── Restaurar nova chave no cluster → ArgoCD sincroniza ✅
```

## Targets no Makefile

| Target | Descrição |
|--------|-----------|
| `make gen-keys` | Gera o par de chaves inicial (executado uma vez no primeiro setup) |
| `make rotate-keys` | Gera novas chaves e re-encripta todos os SealedSecrets automaticamente |

## Fluxo do Dia a Dia (criar novo secret)

```bash
# 1. Criar o secret como dry-run e encriptar com kubeseal
kubectl create secret generic meu-secret \
  --from-literal=chave=VALOR \
  --dry-run=client -o yaml \
  | kubeseal --cert ~/.homelab/sealed-secrets-cert.pem \
  -o yaml > apps/meu-app/sealed-secret.yaml

# 2. Commitar — o arquivo encriptado é seguro para o Git
git add apps/meu-app/sealed-secret.yaml
git commit -m "adiciona secret: meu-secret"

# 3. ArgoCD sincroniza e o Secret real é criado no cluster automaticamente
```

> **Nota:** com `--cert` apontando para a chave pública local, é possível encriptar secrets **sem acesso ao cluster** — útil para preparar secrets antes de um rebuild.

## Backup da Chave Privada

A chave em `~/.homelab/sealed-secrets-key.yaml` é o **único ponto de falha** do sistema. Recomendações de backup:

- Gerenciador de senhas com suporte a anexos (ex: **Bitwarden**, 1Password)
- Pendrive criptografado guardado em local seguro
- **Nunca** commitar no repositório Git

## Notas Técnicas

- A chave é importada como Secret no namespace `kube-system` antes do ArgoCD subir
- `kubeseal` instalado automaticamente pelo `make deps`
